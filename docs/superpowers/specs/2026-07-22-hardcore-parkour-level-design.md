# 《断空试炼》WhiteRoom 关卡设计

## 概述

《断空试炼》是一张高难度但公平的单人跑酷关卡。玩家需要依次完成精准跳跃、节奏机关与综合终冲三段挑战，最终进入塔顶目标区。关卡强调观察、起跳时机与连续操作，并在每个高难段之后提供检查点，避免一次失误导致整关重来。

## 清单与通关条件

- 关卡名：`断空试炼`
- 类型：`reach_zone`
- 难度：`5`
- 预计首次通关时间：`6–9 分钟`
- 建议通关时间：`300 秒`
- 玩家目标：`穿越三重跑酷试炼，抵达塔顶`
- 通关判定：玩家进入塔顶碰撞地面上的 `goal: true` 区域，由 WhiteRoom Shell 判定通关
- 内容分级：`everyone`

关卡不设置强制倒计时。失败只影响当前位置；玩家会回到最近激活的检查点。长按 R 的软重置和暂停菜单返回 Hub 始终由 Shell 保留。

## 空间与视觉语言

关卡是一座悬浮在深色雾空中的白色几何高塔，路线从低处向塔顶逐段盘升。颜色承担明确的玩法语义：

- 白色与浅灰：可安全站立的平台
- 红色与橙红：会令玩家返回检查点的旋转机关
- 青色：检查点与恢复区域
- 金色：终点和最终路线引导

出生平台面积充足，玩家脚下存在静态碰撞体，出生点半径 2 米内没有危险机关。远处的金色终点从前半程即可辨认，用于建立方向感。

## 路线结构

### 第一段：裂空阶梯

玩家从宽阔出生台进入一组逐渐升高的悬空平台。前半段使用 2.0–2.4 米宽平台和 2.3–3.0 米间距，后半段缩窄至 1.5–1.8 米并加入斜向落点，最大水平间距不超过 3.8 米。路线中保留两块低位恢复台，让轻微偏差不必总是坠落。

最后三次跳跃需要连续完成，但落点方向保持可读。段末是一块稳定的青色平台；玩家进入时调用 `sdk.player.setCheckpoint()`，并显示简短提示。

### 第二段：赤轮回廊

玩家依次穿过三座圆形机关平台。每座平台中心都有红色旋转横杆，旋转速度和方向逐步变化：第一座用于识别机制，第二座要求利用空档移动，第三座使用方向相反的双横杆形成连续节奏。

旋转机关是视觉几何体，危险判定在 `onUpdate` 中根据玩家位置、横杆角度、水平距离与高度窗口计算。命中时不结束整关，而是把玩家传送回最近检查点，并显示“被赤轮击中”的提示。三座机关之间放置稳定等待台，玩家可以观察规律后再行动。段末设置第二个青色检查点。

### 第三段：断桥终冲

最终段将前两段技巧组合为一条上升路线：两次窄台斜跳、一座带旋转横杆的中转台、一次高低差跳跃，以及最后五次连续跳跃。平台仍遵守至少 1.4 米宽的可玩下限，连续跳跃中的间距控制在 2.6–3.6 米。

最终平台面积足以安全落地。金色目标区完整位于碰撞地面上，避免玩家穿过却未触发。进入目标区后由 `reach_zone` 的引擎判定完成关卡。

## 运行时结构

`main.js` 默认导出 `createLevel(sdk)`，只使用 Shell 注入的 `sdk`，不导入 Three.js，也不访问浏览器、网络、存储或动态执行 API。内部按职责拆分为以下小型构建函数：

- 材质与共享几何初始化：集中创建并复用几何体和材质
- 静态平台构建：创建网格、加入场景并注册碰撞体
- 路线分段构建：分别生成三段路线与视觉引导
- 检查点注册：通过触发区激活检查点并记录当前位置
- 旋转机关更新：保存机关数据，在 `onUpdate` 中更新角度与危险判定
- 重生处理：坠落由 `killY` 和 Shell 处理；机关命中时传送到当前检查点

`onUpdate(dt, elapsed)` 只更新机关角度和玩家命中检测，不在每帧创建 Geometry、Material、Vector 或闭包。`onDispose()` 不需要清理额外监听器或计时器，因为关卡不创建场景外资源。

## 状态流

玩家进入检查点触发区后，关卡更新当前检查点位置并调用 `sdk.player.setCheckpoint()`。旋转机关每帧先更新视觉角度，再用相同角度计算横杆危险范围。若玩家位于危险范围内且不处于短暂重生保护期，关卡将玩家传送至当前检查点、显示提示，并启动短暂保护期以防连续触发。

终点不调用 `sdk.state.complete()`。它使用 `sdk.helpers.goalZone()` 注册 `goal: true` 区域，通关完全交给 `reach_zone` 的 Shell 判定。

## 公平性与故障处理

- 出生点安全，脚下有碰撞体，2 米内无危险机关。
- 每个高难段后都有检查点，所有检查点位于稳定平台中央。
- 危险机关使用红色高对比材质，并为玩家提供可观察的稳定等待台。
- 机关命中后有短暂保护期，避免在传送帧重复受击。
- 所有跳跃遵守 WhiteRoom 几何关卡建议尺寸，并保留可辨认的前进方向。
- `killY` 设在所有恢复台之下，真实坠落才触发死亡重生。
- 若运行时检测到玩家位置异常低，Shell 仍可依据 `killY` 回到最近检查点。
- 关卡代码不劫持 R 键、暂停菜单或 Hub 返回流程。

## 文件与资源

使用 WhiteRoom Level Kit 脚手架生成基础目录，并编辑以下文件：

- `level.json`：清单、出生点、难度、建议时间、`killY` 与 `reach_zone` 通关条件
- `main.js`：几何场景、检查点、旋转机关、目标区与运行时更新
- `solution.md`：经过真实试玩验证的完整通关路线
- `cover.png`：至少 960×540 的 16:9 关卡封面

关卡只使用 SDK 基础几何体，不需要外部 GLB、纹理或音频资源。

## 验收与测试

1. 运行 WhiteRoom 开发 Shell，确认出生点、相机、碰撞体和路线均正常。
2. 在浏览器中从出生点完整跑到塔顶，并确认由引擎触发通关。
3. 主动测试每个检查点、每种旋转机关、坠落重生和长按 R 重置。
4. 检查 `render_game_to_text()`，确保能描述玩家、检查点、机关与目标状态。
5. 检查浏览器控制台，要求无脚本错误和未处理异常。
6. 根据真实试玩路径更新 `solution.md`。
7. 运行静态校验；修复全部错误后重新执行。
8. 打包为 `.wrlevel`，确认包内四个必需文件齐全且符合预算。

完成标准是：关卡可以从出生点稳定通关，三段难度递进且失败可恢复，静态校验通过，并成功生成可供审核的 `.wrlevel` 包。上传平台不属于本次本地制作范围，除非用户在查看最终 ID 和校验结果后另行明确确认。
